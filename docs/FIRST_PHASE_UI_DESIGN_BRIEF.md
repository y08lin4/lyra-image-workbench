# 第一阶段 UI 设计稿与实施边界

更新时间：2026-06-27  
状态：设计稿草案，所有大页面重构、画布技术选型、保留策略强承诺均需用户确认后再实施  
适用范围：Lyra Image Workbench 第一阶段 UI、信息架构、提示词助手、参考图、创作画布、移动端与视觉一致性

## 1. 决策摘要

本阶段建议采用“设计先行、过渡收束”的路线：不直接推倒现有生成页、提示词助手和创作画布，也不继续小修小补堆入口。先确认一个更清楚的创作主路径，再按确认后的边界分批实现。

推荐默认方案：

1. 第一阶段目标是逻辑闭环和可用性，不追求重效果、复杂动画或专业级节点编辑器。
2. `生成 / 提示词助手 / 创作画布` 可以收束为一个“创作”主入口，但第一步应做过渡版，保留现有生成路径和可回退入口。
3. 提示词助手采用轻会话式设计：体验接近 ChatGPT/Claude，但保留一键应用到生成页的短路径。
4. 参考图必须分清“素材库、预览焦点、本次参考篮、语义绑定”。生成前必须明确展示“将提交 N 张参考图”。
5. 创作画布第一阶段推荐优先保证素材组织、参考图选择、预览和提交摘要；拖入/移动/缩放可作为候选第一批，旋转/连接线/多选需视技术选型和工期确认。
6. 移动端必须有独立布局策略，不能只是把桌面工作台压窄。
7. 主题与视觉先统一 token、按钮、分割线、输入框、状态标签、bottom sheet、sticky 条、弹窗、代码块和 Key 遮罩，再考虑更大的视觉风格重塑。

## 4. 产品与设计原则

Lyra Image Workbench 应该像可靠的创作工具，不像营销页，也不应该像挤压后的后台表格。

第一阶段 UI 原则：

1. 主工作流放在最大、最显眼的位置。
2. 任务状态、任务 ID、来源、消耗次数、参考图提交范围和历史必须明确。
3. 普通用户的最短生成路径不能被高级功能藏深。
4. 管理员能力只对管理员可见，但管理员要容易找到。
5. 先用布局、层级和状态降低认知负担，再考虑装饰。
6. 避免嵌套卡片、重复 1px 边框、大量蓝边按钮和全站硬编码黑色。
7. 桌面和移动端是两套使用姿势，移动端需要独立的信息顺序与操作方式。

## 5. 信息架构设计稿

### 5.1 推荐过渡版

建议第一阶段采用过渡版“创作工作台”：

```text
创作
  - 快速生成
  - 提示词助手
  - 创作画布
结果
广场
我的
API 文档
设置
后台（仅管理员）
更多（移动端承载次级入口）
```

实现边界：

- 可以先把“创作”理解为新的组织方式，不立即删除现有生成页入口。
- 旧的生成路径要能继续完成缺 Key -> 设置 -> 生成/图生图 -> 结果。
- 如果用户暂不确认“创作”收束，下一轮只优化现有 `生成 / 提示词助手 / 创作画布` 的局部体验。
- `结果` 继续作为任务状态、结果消费和历史操作的主视图。
- `广场` 只展示已投稿作品，不承担上传提示词或后台管理功能。
- `我的` 放个人资料、余额、流水、订单、邀请和个人投稿。
- `设置` 只管当前账号 Key、默认生成、2FA、图床等个人配置。
- `后台` 只管系统配置、额度支付、用户管理和流水审计。
- 公开 `/api-docs` 与工作台内 API 文档继续双入口，面向不同使用场景，不视为重复。

### 5.2 不推荐方案

不建议在第一阶段采用以下路线：

- 一次性删除生成页，把所有能力强行塞进创作画布。
- 把提示词助手做成完全独立的聊天产品，导致普通生成路径变长。
- 把画布做成节点编辑器，让普通用户先理解工作流再生成图片。
- 为了“一屏看完”牺牲画布空间、结果预览或移动端可操作性。
- 把设置、我的、后台用户管理合并成一个大账户页。

## 6. 提示词助手设计稿

### 6.1 定位

提示词助手是“帮助用户得到可生成提示词”的轻会话工具，不是第一阶段的全自动创作代理。这里的“轻会话”指交互骨架接近对话式，不承诺多 agent、自动连续创作或完整聊天产品。

推荐定位：

- 用户可以用对话式输入描述需求、追问、让助手改写，但简单生成路径不能被助手流程拉长。
- 助手输出的提示词版本要容易复制、应用到生成、应用到画布。
- 用户不必理解多版本管理，也能把一个结果送去生成。

### 6.2 桌面布局

建议桌面采用三段式层级：

```text
左侧：模式与历史
中间：会话与提示词版本
右侧或底部：应用目标与生成摘要
```

关键控件：

- 模式 segmented：`写提示词 / 图片还原 / 灵感 / 历史`。
- 会话区：用户需求、助手回复、版本卡片。
- 版本卡片操作：`应用到快速生成`、`发送到画布`、`复制`、`继续修改`。
- 应用摘要：目标模型、比例、质量、是否带参考图、预计消耗。

### 6.3 移动布局

移动端不做左右栏压缩。建议：

- 会话内容优先，占据主屏。
- 输入框与发送按钮固定在底部。
- 模式、历史、应用目标放进顶部切换或 bottom sheet。
- 助手输出后的主操作必须贴近输出结果，不让用户滚回顶部找按钮。

### 6.4 第一阶段边界

允许进入第一阶段：

- 轻会话历史。
- 一键应用到快速生成。
- 一键发送到创作画布。
- 输出多个提示词版本。
- 图片还原提示词时绑定真实参考图。

暂缓：

- 多 agent 自动连续生成。
- 复杂版本 diff。
- 自动调参和自动重试策略。
- 跨用户共享私有提示词会话。
- 把提示词助手变成唯一生成入口。

待用户确认：

- 提示词助手主操作放顶部还是底部。推荐默认：移动端靠近输出结果和底部输入区，桌面端可在版本卡片内直接放主操作。

## 7. 参考图规则设计稿

参考图是下一阶段最容易出错的地方，必须先冻结概念。

### 7.1 四个概念

| 概念 | 含义 | 第一阶段 UI 表达 |
| --- | --- | --- |
| 素材库 | 用户可选择的历史图、上传图、结果图、收藏图 | 抽屉、列表或画布素材区 |
| 预览焦点 | 当前正在查看、放大或编辑的一张图 | 高亮边框、预览面板、详情区 |
| 本次参考篮 | 如果提示词没有显式 `@图片`，默认提交的参考图集合 | 缩略图条，展示数量和顺序 |
| 语义绑定 | 提示词中的 `@图片1`、`@最近结果` 等绑定真实图片 ID 的 chip | 输入框内 chip，不是纯文本 |

### 7.2 提交规则

推荐默认规则：

1. 如果提示词中出现 `@图片` chip，只提交被 `@` 提及的图片。
2. 如果提示词中没有 `@图片` chip，提交“本次参考篮”里的图片。
3. 提交前必须展示：`将提交 N 张参考图`，并展示缩略图、顺序和来源。
4. `@图片` 必须绑定真实 uploadId、result image ID 或任务创建时的快照/可复现引用，不能只是普通文本。
5. 创建任务时建议快照参考图来源、顺序、角色和所有者，避免后续保留策略、用户删除或历史变更导致任务不可复现。
6. 第一阶段继续沿用最多 8 张参考图的限制，除非后端和上游能力另行确认。
7. 右键菜单必须有移动端长按或“更多”菜单替代，也要有键盘可达路径。

### 7.3 禁止行为

第一阶段不要做：

- 用户写了 `@图片1`，但系统只把它当普通字符串。
- 用户只预览了一张图，系统却静默提交全部素材库图片。
- 从其他用户空间引用图片。
- 任务创建后仍依赖可变的历史图状态。
- 让外部 URL 直接成为私有参考图来源。

## 8. 创作画布设计稿

### 8.1 定位

创作画布是轻量对象画布，用来组织素材、提示词和参考关系。它不是第一阶段的专业设计软件，也不是节点执行引擎。

推荐目标：

- 用户能把历史结果、上传图和参考图放在一个空间里。
- 用户能通过拖拽、点击菜单或 `@图片` 把图纳入本次生成。
- 用户能看懂哪些图片会被提交，哪些只是素材。
- 用户能在画布中把提示词送到生成任务。

### 8.2 第一阶段能力

允许进入第一阶段：

- 从历史结果、上传图、参考图条拖入画布。
- 画布对象移动、缩放。旋转、多选和连接线建议作为 P1/P2 或技术选型确认后进入。
- 单选和删除；多选待用户确认或后置。
- 右键/长按菜单：`设为参考图`、`插入 @图片`、`复用提示词`、`下载/预览`、`移除`。
- 简单连接线或关系标记作为候选增强，用于表达“参考这张图生成”，需用户确认是否进入第一阶段。
- 画布内明确展示“本次将提交”的参考图集合。
- 移动端提供对象列表、底部工具条或 bottom sheet，不强求完整桌面画布体验。

暂缓：

- 蒙版、局部重绘、图层面板。
- 专业级对齐、吸附、标尺和组件系统。
- 复杂节点图执行。
- 多人协作画布。
- 无限历史分支和完整版本控制。

### 8.3 技术选型与依赖边界

第一阶段默认不新增 Konva、react-konva、Fabric、Pixi 等重画布依赖；优先用现有 React/DOM/CSS/SVG 做轻量画布或交互原型。

如果确认要做高级对象画布，建议单独做技术 spike：验证 `konva + react-konva` 的 React peer 兼容、bundle 体积、懒加载、移动端触控、canvas 可访问性、构建影响和维护成本。未经确认，不把 Konva 作为默认实现前提。

### 8.4 验收标准

- 画布上的“素材”和“本次会提交的参考图”能被用户一眼区分。
- 右键能力在移动端有长按或更多菜单替代。
- 提交前能看到任务摘要、参考图数量、消耗次数和目标模型。
- 画布不是必须一屏塞完，桌面允许大画布，移动端允许聚焦模式和分步操作。

## 9. 结果页与任务详情

结果页继续作为任务和图片消费的主视图。

第一阶段要求：

- 任务列表展示状态、来源、任务 ID、进度、时间和错误状态。
- 当前任务结果优先展示，尤其在移动端。
- 任务详情弹窗保持轻量，只放任务元信息、状态、错误、管理动作和必要参数。
- 完整图片浏览、复用、下载、提交广场、作为参考图等操作留在 `ResultCanvas`。
- API 创建的任务与 Web 创建的任务进入同一个结果历史。
- 未投稿结果显示保留策略提示；已投稿结果只有在复制到广场稳定存储成功后，才显示“广场副本用于展示”。

不要回退到：

- 详情弹窗内再复制一套完整结果画布。
- 队列、结果、详情三个地方重复承担同一组结果操作。
- 移动端先展示长队列，用户要滚很久才看到当前结果。

## 10. 移动端设计稿

当前移动端代码侧已有第一轮收敛，但后续大改仍需遵守移动端独立策略。

### 10.1 导航

当前已完成的移动端主导航可以作为阶段基线：

```text
生成 / 结果 / 广场 / 我的 / 更多
```

如果确认采用“创作”主入口，可在下一轮把 `生成` 调整为 `创作`；未确认前移动端继续显示 `生成`，且创作入口不得让普通用户先进入复杂画布。

### 10.2 页面优先级

移动端推荐优先保证这条路径：

```text
缺 Key
  -> 设置
  -> 保存或返回
  -> 快速生成或图生图
  -> 结果页看到任务和图片
  -> 作为参考图或提交广场
```

移动端具体要求：

- 底部主导航不超过 5 个常驻入口。
- 关键按钮触控目标不小于 44px。
- 结果页当前结果优先，历史队列折叠或放到二级区。
- 设置页 Key 保存动作保持可达，长表单应有 sticky 保存条或分区保存。
- API 文档代码块横向滚动，复制按钮固定可点。
- 画布操作使用 bottom sheet、长按菜单、对象列表和聚焦编辑，不强制桌面画布完整缩小。
- 提示词助手输入框和主操作贴近底部，但不能遮挡输出内容。

## 11. 主题与视觉一致性

本阶段视觉目标是“干净、清楚、可信”，不是做一套新品牌大片。

优先处理：

- 统一 CSS token：背景、表面、文字、弱文字、边框、主色、危险、成功、警告、阴影。
- 普通按钮、主按钮、图标按钮、segmented control、输入框、状态标签统一。
- 降低重复 1px 边框、嵌套卡片和历史蓝边按钮的存在感。
- 绿色、粉色、蓝色、暗色主题下不能残留大块写死黑色或低对比弱文字。
- API Key、上游 Key 等敏感字段默认遮罩，显示/复制动作清楚。
- 加载、失败、空状态要有下一步行动，不只给解释文字。

暂缓：

- 完整品牌视觉重塑。
- 大面积插画、渐变背景和营销页式 hero。
- 重动画和复杂过渡。

## 12. 实施边界

### 12.1 用户确认前禁止

在用户确认设计稿前，不实施以下内容：

- 全局导航大改。
- 合并或删除生成页、提示词助手、创作画布入口。
- 创作画布交互大重构。
- 提示词助手聊天式大改。
- 参考图 payload 破坏性变更。
- 大面积 CSS 主题重写。

### 12.2 第一阶段可实施

设计确认后，可以进入第一阶段实现的内容：

| 方向 | 可实施范围 | 边界 |
| --- | --- | --- |
| 信息架构 | 过渡版创作入口、移动端更多菜单、后台内部 tab 梳理 | 不直接删除旧路径 |
| 提示词助手 | 轻会话、版本卡片、一键应用生成/画布 | 不做全自动创作代理 |
| 参考图 | 参考篮、`@图片` chip、提交前摘要、快照字段设计 | 不跨用户、不静默提交 |
| 创作画布 | 素材拖入、选择、预览、设为参考图、提交摘要优先；移动/缩放按技术选型确认；旋转/连接线后置 | 不做蒙版、图层、节点执行；不默认新增重画布依赖 |
| 结果页 | 结果优先、轻量详情、参考图复用路径 | 不复制完整结果画布到弹窗 |
| 移动端 | 5 入口导航、结果优先、设置 sticky 保存、bottom sheet | 不把桌面三栏硬压到手机 |
| 视觉 | token、按钮、边框、输入框、状态标签降噪 | 不做完整主题系统重写 |

### 12.3 建议拆分

后续如果继续多子代理并行，建议按以下方式避免写入冲突：

1. `requirements-analyst`：冻结用户确认项和验收标准，只读。
2. `ui-ux-reviewer`：桌面/移动交互审查，只读。
3. `frontend-engineer`：单一 UI 实现者，写入指定 `web/src` 文件，避免多人同时碰 `WorkbenchPage.tsx` 和 `styles.css`。

## 14. 需要用户确认的问题

请优先确认这些问题，确认后才能安全进入大页面重构：

| 决策项 | 推荐默认 | 影响 |
| --- | --- | --- |
| 是否采用“创作”主入口收束生成、助手、画布 | 采用过渡版，保留快速生成默认路径 | 决定导航和 Workbench 结构 |
| 提示词助手形态 | 轻会话式，不做唯一入口 | 决定助手页面布局和主操作位置 |
| 提示词助手主操作位置 | 移动端贴近输出和底部，桌面端放版本卡片内 | 影响桌面/移动布局 |
| `@图片` 提交规则 | 有 `@` 只提交被提及图片，无 `@` 提交本次参考篮 | 影响前后端契约和用户预期 |
| 创作画布第一阶段能力 | 素材拖入、选择、预览、设为参考图、提交摘要优先；移动/缩放候选第一批，旋转/连接线/多选待确认 | 影响实现规模 |
| 是否把 `生成` 移动端标签改为 `创作` | 等创作入口确认后再改 | 影响现有用户路径 |
| 视觉降噪优先级 | 先 token 和高频组件，覆盖代码块、Key 遮罩、bottom sheet、sticky 条和弹窗；后完整主题 | 影响 CSS 写入范围 |

| 保留策略文案口径 | 清理器未落地前写成策略提示；真实 30 天清理上线后再写强承诺 | 影响用户信任和数据生命周期 |
| 旋转/连接线/多选 | 是否进入第一阶段，还是放到第二阶段 | 影响画布工期和稳定性 |
| 主题范围 | 触达页面 token 降噪，还是完整主题系统重写 | 影响 CSS 写入范围 |
| 下一轮验收方式 | 本地人工截图加构建检查，能跑浏览器后补 Playwright | 影响交付证据 |
