## 第一阶段重构判断

当前前端“能用”，但不完全符合手机端和普通用户的使用习惯。推荐采用中等重构，不建议小修小补，也不建议推倒重写。

第一阶段建议范围：

1. 移动端导航收敛：底部主入口固定为 `生成 / 结果 / 广场 / 我的 / 更多`。
2. 手机端结果优先：结果页先展示当前结果，任务历史改成可展开区域或 bottom sheet。
3. 设置页移动端补 Key 路径：缺 Key 跳设置保留，但保存动作要始终可达。
4. 后台内部模块化：入口仍只保留工作台 `后台` tab，后台页内部拆成 `总览 / 系统配置 / 额度支付 / 用户管理 / 用户流水` 或待定名称。
5. 任务详情轻量化：结果浏览只留在 `ResultCanvas`，详情弹窗只放任务元信息、状态、错误和管理动作。
6. CSS 视觉降噪：先统一 token 和按钮/segmented 语言，再处理结果区、队列区、创作画布网格。

第一阶段暂不建议做：

1. 不合并生成页和创作画布。
2. 不取消缺 Key 跳设置。
3. 不合并“我的”和后台用户管理。
4. 不移除公开 `/api-docs` 与工作台 API 文档双入口。
5. 不重做完整主题系统。

## 画布想法判断

你的“历史生成图拖到画布、右键设为参考图、提示词里 `@图片1/@图片2` 多图引用”的方向是合理的，和当前已有的“作为参考图”、最多 8 张参考图、创作画布预览区是顺着走的。

必须控制的边界：`@图片1` 不能只是普通文本，必须绑定真实图片 ID 或上传 ID；提交任务时要快照参考图，避免保留策略调整或用户删除后任务失效；右键菜单需要移动端/键盘替代入口；多图引用要限制数量、保留顺序、限定当前账号空间，不能跨用户引用。

## 5 个新增功能候选

### 暂定设计原则

1. 第一阶段目标是逻辑闭环和可用性，不追求重效果。
2. 生成任务状态、任务 ID、参考图提交范围、消耗次数必须在生成前后清楚展示。
3. 提示词助手可以更像 ChatGPT/Claude，但不能把简单生成路径藏深。
4. 创作画布应是轻量对象画布，不是完整 Figma，也不是拥挤节点编辑器。
5. 手机端必须有独立布局策略，不能只是把桌面页面压窄。
6. 主题必须依赖统一 token，不能在绿色/粉色/蓝色主题下仍出现大块黑色默认样式。
