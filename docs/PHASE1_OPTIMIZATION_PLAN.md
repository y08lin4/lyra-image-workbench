# 第一阶段优化计划：逻辑闭环与简洁 UI

更新时间：2026-06-26  
目标分支：`dev`  
阶段定位：把 Lyra Image Workbench 收敛成一个逻辑完整、可验收、可继续商业化的轻量生图站点 MVP。

## 1. 本阶段目标

第一阶段不追求复杂特效、不追求专业级节点编辑器、不继续堆入口。目标是把用户从注册到生成、充值、查看结果、投稿广场、管理员运营这一条主链路做完整，并把 UI 调整到简洁、直觉、好用。

核心闭环：

```text
用户注册/登录
  -> 获得初始或每日免费额度
  -> 选择生成方式并提交任务
  -> 后端扣次数并创建任务
  -> 用户在结果页看任务状态、任务 ID、结果和消耗
  -> 用户可提交结果到广场永久保留
  -> 用户可充值继续使用
  -> 邀请用户首次充值后获得奖励
  -> 管理员可配置上游、额度、支付、用户和流水
```

本阶段完成后，用户应该能自然理解：

- 我现在有没有次数。
- 我能不能生成。
- 生成任务现在到哪一步。
- 这次生成花了多少。
- 我的图会保存多久。
- 我要怎样永久保留到广场。
- 我要怎样充值。
- 管理员在哪里配置站点和处理用户额度。

## 2. 设计原则

1. **主路径优先**：生成、结果、充值、广场、我的、后台，是第一阶段主路径；其他页面都为主路径服务。
2. **少解释，多状态**：用明确状态、按钮、提示和流水说明系统发生了什么，不堆长文案。
3. **普通用户优先**：不要把 ComfyUI 式复杂工作流放到主路径；节点工作流只能作为轻量可视化向导。
4. **账务可追溯**：所有额度变化必须有流水，管理员手动加次数必须有原因。
5. **结果可确认**：任务 ID、来源、状态、错误、重试、消耗、图片保留策略都要明确。
6. **UI 简洁一致**：减少大卡片堆叠、减少嵌套卡片、减少大面积装饰；按钮、表单、列表、状态标签统一。
7. **先单节点稳定**：不做多节点队列、复杂限流、优惠券、套餐体系、提现、多级分销。

## 3. 本阶段范围

### 必做

- 用户注册、登录、邮箱、头像/显示名、我的主页。
- 用户额度余额、消耗流水、管理员加次数流水。
- 免费额度配置：新用户初始额度、每日免费额度。
- 管理员入口只对管理员显示。
- 管理员配置上游 Key、免费额度、易支付、邀请奖励。
- 任务创建走后端 API，前端显示任务状态、任务 ID、来源、历史。
- 生成成功后扣次数或创建任务时扣次数，需要有失败回滚策略。
- 结果页合并任务队列，成为任务和结果的唯一主视图。
- 结果图提交到广场，提交后永久保留；未提交按 30 天策略提示。
- 广场浏览、点赞、每日榜、我的投稿。
- 充值入口、易支付订单、回调入账、订单列表。
- 邀请系统：被邀请用户首次充值成功后，邀请人获得奖励次数。
- API 文档静态页：调用示例和一键复制给 AI 的接入提示词。
- 多主题和暗色模式可读性修复。
- 节点工作流做轻量向导，不做重型拖拽工作流。

### 暂缓

- 多级分销、提现、佣金、发票。
- 复杂套餐、优惠券、会员等级。
- 分布式队列、多节点一致性。
- 精细限流和风控策略。
- 完整邮件验证闭环。
- 专业级 ComfyUI 复刻、自由拖拽节点、复杂工作流保存。
- 把未验收功能合并到 `master`。

## 4. 业务闭环拆解

### 4.1 用户闭环

目标：用户有明确身份、余额、作品、邀请关系和流水。

任务：

1. 注册表单包含用户名、邮箱、密码、邀请码。
2. 登录支持用户名或邮箱。
3. 用户主页展示头像、用户名、显示名、邮箱、余额。
4. 用户主页展示额度流水：生成消耗、充值入账、管理员加次数、邀请奖励、失败回滚。
5. 用户主页展示我的广场投稿。
6. 用户可复制自己的邀请码或邀请链接。
7. 首个注册用户自动成为管理员，后续管理员可设置其他管理员。

验收：

- 新用户注册后能看到余额。
- 用户生成一次后流水减少。
- 管理员给用户加次数后，用户主页能看到原因和管理员。
- 被邀请用户首次充值后，邀请人流水出现奖励。

### 4.2 额度闭环

目标：所有次数变化可追溯，失败不乱扣。

建议模型：

```text
CreditLedgerEntry
  id
  username
  delta
  balanceAfter
  type: initial_free | daily_free | task_charge | task_refund | admin_add | purchase | referral_reward
  reason
  sourceId
  adminActor
  relatedUsername
  createdAt
```

任务：

1. 新用户注册时按管理员配置发放初始免费额度。
2. 每日免费额度按自然日发放，需防重复领取。
3. 创建任务前检查余额是否足够。
4. 任务创建成功后扣减次数，并记录 `task_charge`。
5. 如果任务创建失败且未真正进入上游，回滚次数并记录 `task_refund`。
6. 如果任务部分成功，第一阶段先按创建时数量扣费，不做复杂按成功张数计费，除非现有实现已经支持。
7. 管理员手动加次数必须填写原因。
8. 所有流水只追加，不修改旧记录。

验收：

- 余额不足不能创建任务。
- 同一任务不能重复扣费。
- 同一订单不能重复入账。
- 管理员加次数没有原因时不能提交。

### 4.3 生成任务闭环

目标：用户提交后立刻拿到任务 ID，前端显示状态和历史。

任务：

1. 生成页和节点页都调用统一任务创建 API。
2. 任务创建响应必须包含任务 ID。
3. 前端任务列表展示状态、来源、任务 ID、进度、时间。
4. 任务详情展示模型、比例、质量、格式、消耗次数、错误原因。
5. 结果页通过 SSE 或定时刷新更新状态。
6. 用户不能手动乱刷新任务列表，改为自动定时刷新和状态订阅。
7. API 创建的任务和 Web 创建的任务都进入同一个历史。
8. 重试任务要明确是否再次扣次数。

验收：

- Web 提交任务后跳到结果页，能看到任务 ID。
- API 提交任务后，前端历史能看到来源为 API。
- 任务失败时用户能看到失败原因。
- 任务完成后图片比例不变形。

### 4.4 结果和广场闭环

目标：结果页是生成后的唯一主视图，广场只接收确认投稿。

任务：

1. 合并任务队列和结果页。
2. 结果页左侧或顶部展示任务列表，主区域展示当前任务结果。
3. 每张成功结果图有“提交到广场”按钮。
4. 点击提交时弹确认，展示标题、提示词、模型、比例、质量、图片缩略图。
5. 提交成功后复制图片到广场永久目录。
6. 未提交结果显示“30 天后清理”；已提交结果显示“已永久保留到广场”。
7. 广场只展示已投稿作品，不再提供单独上传提示词入口。
8. 广场图片使用原比例容器，禁止拉伸。
9. 广场支持点赞、取消点赞、每日榜、我的投稿。

验收：

- 未投稿结果能清楚看到 30 天提示。
- 投稿后广场能看到作品。
- 投稿后原任务结果即使未来清理，广场副本仍可访问。
- 每个用户对同一作品只能点赞一次。
- 每日榜只统计当天 00:00-24:00 的投稿作品。

### 4.5 充值和邀请闭环

目标：用户能充值，管理员能配置，系统能幂等入账。

任务：

1. 管理员配置易支付开关、网关、商户 PID、商户 Key、单价、最小充值次数、支付方式。
2. 用户主页或充值页展示充值入口。
3. 用户选择充值次数和支付方式后创建订单。
4. 后端生成易支付跳转链接。
5. 易支付回调校验签名、金额、订单号、状态。
6. 回调成功后给用户加次数并写 `purchase` 流水。
7. 重复回调返回成功但不重复加次数。
8. 如果用户是被邀请用户，且这是首次成功充值，给邀请人发奖励并写 `referral_reward` 流水。
9. 用户主页展示订单列表和状态。

验收：

- 未配置易支付时，用户看不到可用支付或提示“充值暂未开放”。
- 创建订单后有订单号和支付链接。
- 签名错误不会入账。
- 金额不匹配不会入账。
- 重复回调不重复加次数。
- 邀请奖励只发一次。

### 4.6 管理后台闭环

目标：管理员不需要拼 URL，能完成站点运营最小动作。

任务：

1. 管理员用户前端显示“管理后台”入口。
2. 普通用户不显示后台入口，直接访问后台也要被后端拒绝。
3. 后台配置上游 Key 或上游 Base URL。
4. 后台配置新用户初始免费额度。
5. 后台配置每日免费额度。
6. 后台配置易支付。
7. 后台配置邀请奖励次数。
8. 后台用户列表展示邮箱、余额、管理员状态、注册时间。
9. 后台可给用户加次数，原因必填。
10. 后台可查看用户流水。
11. 后台可设置或取消管理员，需要确认。

验收：

- 普通用户看不到后台入口。
- 管理员能从前端直接进入后台。
- 修改免费额度后新注册用户生效。
- 管理员加次数后用户余额和流水同步更新。

## 5. UI 优化计划

第一阶段 UI 目标是“少但清楚”，不要重视觉，不要复杂动画。

### 5.1 导航结构

建议一级导航保留：

```text
生成
提示词助手
节点工作流
提示词库
广场
结果
我的
API 文档
设置
后台（仅管理员）
```

优化点：

1. “结果”承担任务队列和结果详情，不再额外出现队列入口。
2. “提示词助手”独立一级页面，不再挤在生成页。
3. “节点工作流”定位为可视化向导，不要替代生成页。
4. “设置”只放用户 Key、本地偏好、主题等用户级配置。
5. “后台”只给管理员显示，放系统级配置。

### 5.2 生成页

目标：最快完成一次生成。

优化点：

1. 主输入区更宽，提示词框足够大。
2. 参数区只展示必要项：模式、模型、比例、清晰度、质量、格式、数量。
3. Key 缺失时用明确提示和“去设置”按钮。
4. 提交按钮显示预计消耗次数。
5. 提交后直接跳到结果页。
6. 不在生成页塞提示词助手完整功能。

验收：

- 用户 10 秒内能知道在哪里写提示词、在哪里提交。
- 页面没有多层卡片套卡片。
- 移动端提交按钮不会被挤变形。

### 5.3 节点工作流页

目标：让普通用户理解生成流程，不做专业节点编辑器。

优化点：

1. 保持大画布平铺。
2. 上方是步骤和模板，中间是流程画布，下方是当前节点参数。
3. 不做复杂拖拽连线。
4. 点击节点只用于切换参数区域。
5. 文案不要讲“功能介绍”，只讲当前节点要填什么。
6. 保证 1280、1440、1920 宽度下流程不堆叠。

验收：

- 一眼能看到从输入到结果的完整链路。
- 没有左右栏把画布挤成一团。
- 移动端允许画布横向滚动，但整页不乱横滚。

### 5.4 结果页

目标：任务和图片都在一个地方处理。

优化点：

1. 左侧任务列表，右侧当前任务详情；移动端上下排列。
2. 任务列表显示状态、进度、来源、任务 ID 短码。
3. 详情顶部显示任务 ID、来源、模型、消耗、创建时间。
4. 图片区域按原比例展示，不裁切、不拉伸。
5. 每张图提供下载、复制、作为参考图、提交广场。
6. 错误状态给出失败原因和重试入口。

验收：

- 用户不用切换页面就能看队列和结果。
- 图片不变形。
- API 任务和 Web 任务都能看见。

### 5.5 广场页

目标：像作品流，不像后台列表。

优化点：

1. 移除上传提示词入口。
2. 顶部只保留筛选：最新、每日榜、我的投稿。
3. 作品卡片展示图片、标题、作者、点赞、模型/比例标签。
4. 图片容器按比例自适应，不强制统一裁切。
5. 每日榜强调排名和点赞数，不做复杂视觉。
6. 点赞按钮即时反馈。

验收：

- 所有图片不拉伸。
- 用户能清楚区分最新投稿和每日榜。
- 我的投稿能只看自己的作品。

### 5.6 我的页面

目标：账户中心。

优化点：

1. 顶部显示头像、显示名、用户名、邮箱、余额。
2. 明确展示邀请码和复制按钮。
3. 展示额度流水。
4. 展示充值订单。
5. 展示我的广场投稿。
6. 流水支持按类型筛选可以放到第二阶段，本阶段先按时间倒序。

验收：

- 用户能看到自己还剩多少次数。
- 用户能看懂每次次数变化来自哪里。

### 5.7 管理后台

目标：运营动作清楚，不追求花哨。

优化点：

1. 后台首页分区：系统配置、额度配置、支付配置、用户管理。
2. 配置项用表单，不用复杂卡片。
3. 保存成功/失败要有明确提示。
4. 用户列表支持搜索用户名/邮箱。
5. 用户加次数使用小表单或弹窗，原因必填。
6. 敏感配置只显示是否已设置和掩码，不显示明文。

验收：

- 管理员能在 1 分钟内找到加次数入口。
- 管理员能看到某个用户的流水。
- 普通用户无法访问。

### 5.8 主题和暗色

目标：所有主题可读，不追求复杂品牌感。

优化点：

1. 统一 CSS 变量：背景、表面、文字、弱文字、边框、主色、危险、成功、警告。
2. 输入框、按钮、标签、弹窗、表格统一使用变量。
3. 暗色模式先保证对比度，不用过多渐变。
4. 主题选择放在全局顶部。
5. 新页面必须只用变量，不写死浅色文本。

验收：

- 白蓝、黑紫、白紫、绿色、粉色、蓝色主题下，主要文字都能看清。
- 输入框 placeholder 和弱文字不发灰到看不见。

## 6. 建议实施顺序

### Wave 1：数据和接口先收口

1. 用户资料、邮箱、管理员角色。
2. 额度余额和流水。
3. 免费额度配置。
4. 任务扣费和失败回滚策略。
5. 管理员加次数。
6. 易支付订单和回调。
7. 邀请首充奖励。
8. 广场投稿、点赞、每日榜后端。

完成标准：

- 后端测试能覆盖额度、订单、邀请、广场投稿。
- 所有核心接口能编译通过。

### Wave 2：主路径前端

1. 我的页面账户中心。
2. 生成页简化。
3. 结果页合并任务队列。
4. 广场页重构。
5. 后台用户和配置页。
6. 充值入口和订单列表。
7. API 文档静态页。

完成标准：

- `npm run build` 通过。
- 主要页面在 1280、1440、移动端不变形。

### Wave 3：UI 统一和体验整理

1. 统一按钮、输入框、状态标签。
2. 清理多余说明文案。
3. 暗色和多主题修复。
4. 节点工作流轻量化调整。
5. 图片比例和预览体验复查。
6. 空状态、错误状态、加载状态补齐。

完成标准：

- 用户主路径不需要读长说明。
- 失败和缺配置状态都有下一步按钮。

### Wave 4：验收和风险收口

1. 全量后端测试。
2. 前端构建。
3. 残留视频/MiniMax 扫描。
4. 敏感字段扫描。
5. 本地启动验收。
6. 真实上游生成测试。
7. 模拟或真实易支付回调测试。
8. 写变更说明。

完成标准：

- `go test ./...` 通过。
- `npm run build` 通过。
- 本地 `http://127.0.0.1:5173/` 可完整走主路径。

## 7. 子任务拆分建议

如果继续使用多代理并行，建议这样拆，避免写同一个文件互相覆盖。

| 子任务 | 类型 | 写入范围 | 目标 |
| --- | --- | --- | --- |
| A1 用户和额度 | 后端 | `internal/users/*`, `internal/api/users.go`, `internal/api/admin_users.go` | 用户资料、余额、流水、管理员加次数 |
| A2 支付和邀请 | 后端 | `internal/billing/*`, `internal/api/billing.go`, `internal/settings/*` | 易支付订单、回调、邀请首充奖励 |
| A3 任务扣费 | 后端 | `internal/jobs/*`, `internal/api/tasks.go` | 创建任务扣费、失败回滚、任务来源 |
| A4 广场后端 | 后端 | `internal/promptsquare/*`, `internal/api/prompt_square.go` | 结果投稿、点赞、每日榜、我的投稿 |

| B2 结果页 | 前端 | `ResultCanvas`, `TaskSidebar`, `WorkbenchPage` | 队列结果合并、提交广场 |

## 8. 验收路径

### 8.1 用户主路径

1. 注册新用户。
2. 确认获得初始免费额度。
3. 写提示词生成 1 张图。
4. 跳转结果页，看到任务 ID、状态、消耗次数。
5. 图片生成成功，不变形。
6. 提交到广场。
7. 广场看到作品。
8. 点赞作品。
9. 用户主页看到作品和流水。

### 8.2 管理员路径

1. 管理员登录。
2. 顶部出现后台入口。
3. 配置上游 Key。
4. 配置新用户初始免费额度。
5. 配置每日免费额度。
6. 给某个用户加次数并填写原因。
7. 查看该用户流水。
8. 配置易支付。
9. 配置邀请奖励。

### 8.3 支付路径

1. 用户进入我的页面。
2. 选择充值次数。
3. 创建订单。
4. 跳转或复制支付链接。
5. 模拟成功回调。
6. 余额增加。
7. 订单状态变为成功。
8. 重复回调不重复增加。
9. 如果用户有邀请人，邀请人只获得一次奖励。

### 8.4 API 路径

1. 管理员或用户生成 Bearer API Key。
2. 用 curl 创建任务。
3. 返回任务 ID。
4. SDK 轮询任务状态。
5. 前端结果页也能看到 API 来源任务。
6. API 文档复制给 AI 的提示词可用。

### 8.5 UI 路径

1. 1280 宽度检查生成、结果、广场、我的、后台。
2. 1440 宽度检查节点工作流是否平铺。
3. 手机宽度检查导航和按钮是否变形。
4. 切换白蓝、黑紫、白紫、绿色、粉色、蓝色主题。
5. 暗色模式下检查输入框、弱文字、状态标签、弹窗。

## 10. 风险和处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 支付重复回调 | 重复加次数 | 订单号和三方交易号幂等，重复回调只返回成功 |
| 任务失败扣费争议 | 用户体验差 | 明确扣费时机，未提交上游失败必须回滚 |
| 广场永久图丢失 | 投稿失效 | 投稿时复制到永久目录，不引用临时结果文件 |
| 管理员权限泄露 | 普通用户看到后台能力 | 前后端都校验管理员，后端为准 |
| 主题写死颜色 | 暗色看不清 | 新 UI 只使用主题变量 |
| 页面入口太多 | 用户迷路 | 结果合并队列，提示词助手独立，视频入口删除 |
| 节点工作流过重 | 普通用户难懂 | 定位为轻量向导，不做复杂拖拽 |
| 邀请奖励滥发 | 成本不可控 | 只在首次成功充值后奖励一次 |

## 11. 第一阶段完成定义

第一阶段完成不是“页面都看起来有了”，而是下面这些事情可以连续跑通：

1. 用户注册后获得免费额度。
2. 用户能生成图片，任务进入结果页。
3. 任务状态、任务 ID、来源、消耗次数清楚。
4. 用户余额和流水正确变化。
5. 用户能把结果提交到广场永久保留。
6. 用户能点赞广场作品并看到每日榜。
7. 用户能充值，支付成功后次数到账。
8. 邀请奖励只在首充成功后发放。
9. 管理员能配置免费额度、上游、支付和邀请奖励。
10. 管理员能给用户加次数，原因进入流水。
11. API 文档能指导 AI 或开发者只用 Bearer Key 创建任务并轮询。
12. 主要页面在浅色、暗色和多主题下可读、简洁、不变形。

## 12. 建议验收优先级

如果时间有限，按这个顺序验收：

1. 注册、登录、管理员入口。

4. 额度流水。
5. 管理员加次数。
6. 广场投稿。
