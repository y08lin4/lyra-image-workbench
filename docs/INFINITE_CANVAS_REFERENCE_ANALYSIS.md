# basketikun/infinite-canvas 画布参考分析

更新时间：2026-06-28

本文用于把 `basketikun/infinite-canvas` 的只读参考结论收拢成 Lyra 后续画布和 Agent 创作模式的上下文材料。本文只做产品和架构分析，不代表要复制上游代码，也不改变当前 Lyra 的实现边界。

## 1. 总结论

`basketikun/infinite-canvas` 的价值不在于某个单点 UI，而在于它把“画布对象、节点关系、AI 生成、助手、提示词库、本地 Agent/MCP”放在同一个空间里，让用户用空间关系表达创作意图。Lyra 可以借鉴这种“画布即语义输入”的方向，但不能照搬它的运行模式。

对 Lyra 的关键启发：

- 创作画布应该继续作为主入口，而不是只做上传参考图的附属面板。
- 图片、文本、连线、角色和顺序要被整理成可执行的生成上下文，不能只停留在视觉摆放。
- Agent 创作模式应保持独立模块：负责多轮计划、确认和任务闭环，不直接写画布坐标、缩放、旋转、连线状态。
- 画布和 Agent 之间只通过显式动作互通：发送方案到画布、把画布/结果图片作为 Agent 参考、把结果继续修改。
- 上游的本地 Canvas Agent / MCP 思路适合做后续 P1/P2 的“AI 操作画布”能力，但不应成为本轮 Agent 创作模式上线的 P0 依赖。

## 2. 参考项目事实快照

公开仓库：`https://github.com/basketikun/infinite-canvas`

已确认的上游口径：

- 项目定位是面向 AI 图片创作的开源无限画布工作台，包含画布编排、AI 图片生成、参考图编辑、对话助手、提示词库和素材管理。
- 仓库 README 明确项目仍处于开发阶段，不保证历史数据兼容，更适合个人或本地部署，不建议直接公网多人共用。
- 技术栈以 Next.js、React、TypeScript、Tailwind CSS、Ant Design、Zustand、TanStack Query 为主。
- 画布项目、素材和 AI API Key 主要保存在浏览器本地；AI 请求默认由前端浏览器直连用户配置的 OpenAI 兼容接口。
- 许可证为 AGPL-3.0。Lyra 只能吸收产品/交互启发，不能未经评估复制上游代码或文件结构。

## 3. 功能结构观察

### 3.1 画布项目

上游支持多画布项目、重命名、删除、批量选择、JSON 导入导出。项目数据当前主要保存在浏览器本地，不随账号自动同步。

对 Lyra 的启发：

- Lyra 已经是服务端账号/空间模型，不能照搬本地存储。画布草稿应挂到当前 space/user，并明确清理、权限和跨设备策略。
- JSON 导入导出可以作为后续迁移/备份能力，但 P0 更重要的是保存任务引用快照和可复现上下文。

### 3.2 无限画布交互

上游画布支持拖动画布、滚轮缩放、小地图、背景切换、框选、多选、复制粘贴、撤销重做、删除选中、节点连线和上下游高亮。

对 Lyra 的启发：

- 当前 Lyra 的画布能力应优先保证“桌面端可用、同屏不挤压、参考关系能被提交前确认”。
- 小地图、全量快捷键、复杂批量编辑属于增强项，不应阻塞 Agent 创作模式上线。
- 撤销重做如果后续实现，要覆盖节点、连线、视口和参考篮变化，不能只撤销 DOM 位置。

### 3.3 节点模型

上游主要有图片节点、文本节点、生成配置节点。生成配置节点会汇总上游文本和图片，统一设置模型、比例、数量后批量生成。图片节点会记录提示词、生成类型、模型、尺寸、质量、数量和参考图引用；参考图丢失时会提示，避免误发空的图生图请求。

对 Lyra 的启发：

- Lyra 的画布不必完整复刻“生成配置节点”，但需要形成等价的结构化提交模型：文本块、参考图、连线说明、参考图角色、顺序、比例、模型、数量。
- 生成任务必须保存 reference snapshot。历史图被清理或用户删除后，任务仍应知道当时使用了什么引用，不能只保存 `@图1` 文本。
- Agent 的 `AgentReference` 和画布的参考篮可以后续共享 token 编码，但本轮不要求共享内部状态。

### 3.4 图片工作流

上游支持上传、拖入图片、替换图片节点、下载、保存到素材、裁剪、本地变换、失败重试、多图批量结果组和主图切换。

对 Lyra 的启发：

- P0 应收紧到：上传/历史图加入画布、作为参考图、显示顺序和来源、生成前确认。
- 裁剪、局部编辑、主图组管理可以后置；不要在 Agent 创作模式上线时扩大画布范围。
- 失败重试必须沿用原任务参数和引用快照，不能从当前画布状态重新猜。

### 3.5 画布助手

上游右侧助手能读取当前选中节点和上游节点，支持文本问答、生图、粘贴图片、历史会话、重试回答，并能把生成的文本或图片插回画布。

对 Lyra 的启发：

- 这类助手更接近“画布内 Copilot”，和 Lyra 本轮 Agent 创作模式不是同一个 P0。
- Lyra Agent 的主路径是“对话 -> 计划 -> 确认 -> 创建任务 -> 结果 -> 继续修改”，不应直接变成画布侧边栏自动改图工具。
- 后续如果做画布内助手，应先只读画布快照，再由用户确认写入，避免 Agent 擅自移动或删除用户素材。

### 3.6 本地 Canvas Agent / MCP

上游提供本地 Canvas Agent，用来连接网页画布和本机 Codex / Claude Code。它默认监听 `127.0.0.1`，用 token 和 Origin 绑定做本地访问控制；MCP 工具包括读取画布状态、读取选区、导出快照、应用操作、创建文本节点、创建图片提示词流程等。Codex 插件可以自动注册 MCP 并尝试打开本地画布。

对 Lyra 的启发：

- MCP 工具是很好的 P1/P2 方向：让 AI 读取当前画布、提出修改、创建文本节点或生成流程。
- 本轮不应把 MCP 作为上线前置。Lyra 是 SaaS/自部署站点，涉及登录、额度、上游 Key、任务队列和数据权限，不能让本地 Agent 绕过后端任务链路。
- 如果未来做 Lyra Canvas Agent，所有写操作都必须由页面二次确认，并且必须进入现有任务、日志、额度和审计体系。

## 4. 与 Lyra 的关键差异

| 维度 | infinite-canvas | Lyra 当前目标 |
| --- | --- | --- |
| 部署定位 | 个人/本地优先，开发阶段，不建议直接公网多人共用 | 可自部署的账号化生图站点 |
| AI 请求 | 浏览器本地 Key 前台直连 OpenAI 兼容接口 | 后端统一任务链路，支持用户 Key 和系统 Key 额度规则 |
| 数据存储 | 画布/素材/API Key 主要本地保存 | 用户、空间、任务、结果、广场、额度、流水服务端持久化 |
| 画布角色 | 完整 AI 创作工作台 | 主入口，但必须服务任务闭环和 SaaS 权限 |
| Agent 形态 | 画布助手 + 本地 Canvas Agent/MCP | 独立 Agent 创作模块，多轮计划和任务编排 |
| 风险边界 | 适合可信本机环境 | 需要防越权引用、扣费误触发、敏感 Key 暴露 |

## 5. Lyra 可借鉴清单

可以直接吸收为产品原则：

- 画布里的对象关系要进入生成上下文，尤其是图片角色、连线语义和上游/下游关系。
- 生成前给用户看“将提交哪些参考图、顺序、角色、提示词摘要、比例、数量和预计消耗”。
- 图片节点或任务结果应保存生成元数据，方便重试、复用和继续修改。
- 参考丢失要明确提示，不应静默降级成空参考图请求。
- 画布助手或 Agent 生成的结果应能插回当前创作上下文，但必须是显式动作。

不建议本轮吸收：

- 不引入 Next.js/Ant Design/Zustand 作为 Lyra 当前技术栈调整理由。
- 不复制前台直连 Key 模式；Lyra 必须保留后端额度、任务、日志和空间隔离。
- 不把视频生成能力带回 Lyra 当前范围；本轮产品已明确删除 video/MiniMax，GIF 独立建设。
- 不把本地 MCP/插件作为 P0；它适合后续高级工作流，不适合本轮 Agent 创作上线。
- 不引入完整节点编辑器复杂度来阻塞 Agent 的计划/确认/任务闭环。

## 6. 对 Agent 创作模式的落地映射

Agent 创作模式应借鉴上游“AI 理解画布上下文”的思想，但落到 Lyra 时要变成后端可审计的任务编排：

1. `AgentReference` 保存来源快照：upload、task-result、agent-result、canvas-export。
2. `AgentRound` 保存本轮用户消息、引用 token、模型计划、确认状态和 task id。
3. 画布发送到 Agent 时，只发送结构化摘要和引用快照，不发送可变的内部坐标状态。
4. Agent 返回计划时，页面显示目标、场景、参考图用途、参数和预计消耗，而不是把大 textarea 当核心。
5. 用户确认后才调用现有 background task service 创建任务。
6. 任务完成后，结果图可成为下一轮 Agent 参考，也可显式发送到创作画布。

## 7. 画布验收补充

后续如果按参考项目继续强化 Lyra 画布，至少要验收：

- 桌面 1366x768 和 1920x1080 下，画布、输入区、参数、参考篮和生成按钮同屏可用。
- 移动端 390x844 下，上传/历史图加入、查看参考图、删除引用、提交前确认不遮挡。
- 外部图片拖入、历史图加入、粘贴图片、右键/长按设为参考图均能形成真实引用 ID。
- 提交前能看到参考图数量、顺序、来源、角色和连线说明。
- 参考图被删除或清理后，重试/继续修改显示“引用已失效”，不崩溃，不误发空参考。
- 绿色主题、暗色主题下选中态、连线、按钮和状态标签对比度正常。

## 8. 风险与边界

- 许可证风险：上游为 AGPL-3.0，Lyra 本轮只记录思路和公开功能分析，不复制代码。
- 架构风险：上游浏览器本地存储/前台直连 Key 与 Lyra 服务端任务/额度体系不同，不能混用。
- 范围风险：上游包含视频、音频、本地 Agent、插件等更宽功能，Lyra 当前范围要继续控制。
- 验收风险：上游也明确移动端触控体验未系统完善，Lyra 不能只因参考项目支持无限画布就宣称移动端已可用。
- 安全风险：未来若做本地 Agent/MCP，需要 token、Origin、二次确认、审计日志和最小权限工具集。
